# Host capability surfaces: Codex and OpenCode

Research snapshot: **2026-07-10**. Documentation was checked against sandboxed local CLIs: **Codex CLI 0.144.1** and **OpenCode 1.17.18**.

## Executive answer

The hosts overlap at the product-concept level, but not at the lifecycle or storage-contract level. A Matty capability pack therefore needs a **Matty-owned resource vocabulary plus one adapter per host**; it must not treat either host's plugin format as the universal pack format.

The one strong shared projection is agent skills: both hosts discover user skills under `~/.agents/skills/<name>/SKILL.md`. MCP servers also share a useful logical model (named local or remote servers with enablement), but their config schemas and lifecycle commands differ. Every other requested surface needs host-specific projection.

## Lifecycle conclusions

1. **Representation is not ownership.** Most resources are discovered from files/config. Matty must record which exact paths, config fragments, and external setup actions it owns, including shared-resource references across active packs.
2. **Enablement is host-specific.** Codex plugins, skills, hooks, and MCPs each use different toggles/trust; OpenCode mixes permission filters, `enabled`/`disable` fields, presence-based loading, and the all-plugins `--pure` runtime flag.
3. **Updates are mostly reconciliation.** Apart from package installation and marketplace refreshes, neither host exposes a complete update API for every surface. Matty adapters should reconcile desired content/config rather than assume a common host update command.
4. **Removal must be surgical.** Delete only Matty-owned files or config members, preserve user/other-pack content, treat credentials/connectors separately, and retain shared resources until no active pack requires them.
5. **Activation takes effect at different boundaries.** Several resources require a new session or restart; hooks may also require explicit trust. An adapter must report pending restart/trust/auth work rather than call the pack fully usable too early.

## Current Matty seam

CodeGraph showed that current Matty behavior already contains narrow but pre-pack projections:

- `internal/cli/paths.go` resolves shared user skills at `~/.agents/skills`, Codex guidance at `~/.codex/AGENTS.md`, and OpenCode config/guidance under `$XDG_CONFIG_HOME/opencode`.
- `internal/prompt/prompt.go` owns marker-based Codex `AGENTS.md` write/inspect/remove behavior.
- `internal/opencode/opencode.go` owns OpenCode instruction-file/config merge behavior.
- `internal/cli/plan.go` composes skill links, delegated Engram setup, and both prompt projections.

These are useful adapter primitives, but they are not evidence that host lifecycles are equivalent. The capability-pack design should keep discovery/state in Matty-owned modules and adapt per host rather than expand `internal/cli` into the owner of every resource format.

## Sandboxed CLI verification

All mutations used temporary directories and removed them afterward. No operator config was changed.

- Exported temporary `HOME`, `XDG_CONFIG_HOME`, `XDG_CACHE_HOME`, and `XDG_DATA_HOME` where applicable; set temporary `CODEX_HOME` for Codex.
- Explicitly unset `OPENCODE_CONFIG`, `OPENCODE_CONFIG_CONTENT`, and `OPENCODE_CONFIG_DIR`. **Gotcha:** this environment already had `OPENCODE_CONFIG` set, so changing only `HOME`/`XDG_CONFIG_HOME` still caused OpenCode to read that external config. Re-running with the override unset isolated discovery and writes.
- Codex `mcp add` created a temporary `~/.codex/config.toml`; `list/get` showed the enabled server and redacted its environment value; `remove` removed the table.
- Codex `plugin --help` confirmed `add/list/remove/marketplace`; `mcp --help` confirmed `add/list/get/remove/login/logout`.
- OpenCode `mcp add matty-test --url https://example.invalid/mcp` wrote the temporary XDG `opencode.jsonc`; isolated `mcp list` saw only that failed test endpoint.
- OpenCode `plugin --help` confirmed install plus global/force flags. A delegated sandbox check installed a test npm plugin into temporary config/cache and observed the config entry; exact cache layout differed from the documentation, so cache paths are not treated as a stable Matty contract.
- OpenCode `agent create` wrote an agent Markdown file only beneath the supplied temporary path. Agent generation invoked model-backed work, so the documented file format—not generated prose—is the stable integration target.

## Sources and evidence quality

- Product behavior: current official OpenAI/Codex and OpenCode documentation linked in the matrix.
- OpenCode hook contract: first-party `anomalyco/opencode` source linked above.
- CLI/filesystem behavior: sandboxed local commands against the versions stated at the top.
- Matty architecture: current repository source discovered with CodeGraph, then checked through the cited files.
- Filesystem deletion/update statements without an explicit host command are treated as adapter operations or inferences, not as a promised host lifecycle API.
